### 确定的方案

**架构：Peter 模式（非 PR 模式）**
- 所有 agent 在 main 分支工作
- 不用 worktree，不用 PR
- 原子 commit 作为安全网（Conventional Commits）
- 回滚靠 git revert

**Agent 分工：**
- Codex → 后端代码（主力 coder）
- Claude Code → 前端代码（所有前端由 CC 写）
- Claude Code → 规划（cc-plan）
- Claude Code → 代码审查（cc-review）

**并行策略：B 方案 — 文件隔离并行**
- 不同 agent 改不同目录，可以同时跑
- 例如：codex 写后端 + CC 写前端，同时进行
- 我负责确保文件不重叠

**tmux 会话布局（5 基础 agent）：**
- cc-plan: 规划和任务拆解
- codex-1: 后端编码
- cc-frontend: 前端编码
- cc-review: 审查 Codex 写的代码
- codex-review: 审查 CC 写的代码
- 复杂项目可扩展编码 agent（codex-2, cc-frontend-2 等）

**监控：Cron + 我判断**
- Cron 脚本每 3 分钟检查 tmux 状态
- 检测到变化发消息给我
- 我读 tmux 产出做决策

**任务管理：**
- 我拆任务（不是 agent 拆）
- 每个任务 scope 明确，写死文件范围和禁止事项
- commit 检查：git diff --stat 验证改动范围

### 7 个问题（全部已确定）

1. **Git 工作流** — GitHub 远程 repo，每个 commit 立即 push，Conventional Commits
2. **权限边界** — 需求不明确/密钥/多次卡住/架构大改 → 问爸爸，其他我自己决定
3. **任务持久化** — active-tasks.json 放 workspace/swarm/，含 id/name/agent/status/note/commits/issue/depends_on
4. **Prompt 模板** — 固定格式，我出初版，使用中迭代
5. **Review 标准** — 无 critical/high 就通过，不通过返回原 agent 改，3 遍不过换另一个 agent
6. **失败重试** — 原 agent 最多 3 次，失败后换 agent 再试 2 次，还不行通知爸爸
7. **项目环境** — GitHub 已有 repo，Codex Business 可访问，本地无依赖，暂无测试脚本
